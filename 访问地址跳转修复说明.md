# 🔧 访问地址跳转问题修复说明

## 问题描述

用户在控制台点击访问地址时，会被跳转回管理面板，而不是正常访问 SillyTavern 实例。

## 问题原因

1. **Nginx 权限验证机制**  
   当用户访问 `/{username}/st/` 路径时，Nginx 会发起 `auth_request` 到 `/api/auth-check/verify/{username}` 进行权限验证。

2. **Token 传递问题**  
   权限验证需要从 Cookie 中读取 `st_token`，但之前的登录逻辑只将 Token 存储在 `localStorage` 中，没有设置 Cookie。

3. **验证失败导致重定向**  
   因为没有 `st_token` Cookie，Nginx 认证失败（返回 401/403），触发 `@access_denied` 错误处理，导致用户被重定向到拒绝访问页面或管理面板。

## 修复方案

### ✅ 修改的文件

1. **`public/js/auth.js`**
   - 添加 `setCookie()` 函数
   - 在登录成功时设置 `st_token` Cookie
   - 在注册成功时设置 `st_token` Cookie

2. **`public/js/dashboard.js`**
   - 添加 `setCookie()` 和 `deleteCookie()` 函数
   - 在页面初始化时设置 `st_token` Cookie（防止刷新后丢失）
   - 在退出登录时清除 `st_token` Cookie

3. **`public/js/setup.js`**
   - 添加 `deleteCookie()` 函数
   - 在退出登录时清除 `st_token` Cookie

4. **`public/js/admin.js`**
   - 添加 `setCookie()` 和 `deleteCookie()` 函数
   - 在页面初始化时设置 `st_token` Cookie
   - 在退出登录时清除 `st_token` Cookie

### 🔑 核心修复逻辑

```javascript
// 设置 Cookie
function setCookie(name, value, days = 365) {
    const expires = new Date();
    expires.setTime(expires.getTime() + (days * 24 * 60 * 60 * 1000));
    document.cookie = `${name}=${value}; expires=${expires.toUTCString()}; path=/; SameSite=Lax`;
}

// 删除 Cookie
function deleteCookie(name) {
    document.cookie = `${name}=; expires=Thu, 01 Jan 1970 00:00:00 UTC; path=/; SameSite=Lax`;
}
```

**在登录时：**
```javascript
localStorage.setItem('token', data.token);
setCookie('st_token', data.token);  // 新增
```

**在退出时：**
```javascript
localStorage.removeItem('token');
deleteCookie('st_token');  // 新增
```

**在页面加载时：**
```javascript
const token = localStorage.getItem('token');
if (token) {
    setCookie('st_token', token);  // 新增
}
```

## 修复效果

1. ✅ 用户登录后，Token 同时存储在 localStorage 和 Cookie 中
2. ✅ 用户刷新页面后，Cookie 会自动从 localStorage 恢复

4. ✅ 用户点击访问地址后，能够正常访问 SillyTavern 实例
5. ✅ 用户退出登录后，Cookie 会被正确清除

## 技术细节

### Cookie 配置说明

```javascript
document.cookie = `st_token=${token}; expires=${expires}; path=/; SameSite=Lax`;
```

- **`st_token`**: Cookie 名称，与 `routes/auth-check.js` 中读取的名称一致
- **`expires`**: 设置为 365 天，与 Token 有效期匹配
- **`path=/`**: Cookie 在整个域名下有效
- **`SameSite=Lax`**: 防止 CSRF 攻击，同时允许同站导航

### Nginx 权限验证流程

```
用户访问 /{username}/st/
    ↓
Nginx auth_request → /api/auth-check/verify/{username}
    ↓
读取 st_token Cookie
    ↓
验证 JWT Token
    ↓
检查用户权限
    ↓
✅ 通过 → 允许访问 ST 实例
❌ 失败 → 返回 401/403
```

## 常见问题

### Q: 为什么要同时使用 localStorage 和 Cookie？

A: 
- **localStorage**: 用于前端 API 请求的认证（通过 `Authorization: Bearer {token}` header）
- **Cookie**: 用于 Nginx `auth_request` 的权限验证

两者配合使用，确保认证在不同场景下都能正常工作。

### Q: 如果已经登录，需要重新登录吗？

A: 是的。需要重新登录一次，以便设置 `st_token` Cookie。或者刷新控制台页面也可以（初始化代码会自动设置 Cookie）。

### Q: Cookie 什么时候过期？

A: Cookie 设置为 365 天后过期，与 JWT Token 的有效期一致。
