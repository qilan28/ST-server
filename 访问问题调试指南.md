# 🔍 访问问题调试指南

当用户点击访问地址被跳转回管理面板时，可以通过以下步骤进行问题排查。

## 📋 调试步骤

### 1️⃣ 清理环境

```bash
# 在浏览器中按 Ctrl+Shift+Delete
# 清除所有 Cookie 和缓存
```

### 2️⃣ 打开开发者工具

1. **浏览器控制台**：按 F12 或右键 → 检查
2. **切换到 Console 标签页**
3. **保持控制台打开**，不要关闭

### 3️⃣ 重新登录

1. 访问管理平台首页
2. 输入用户名和密码
3. 点击登录

**检查控制台输出**：应该看到类似以下内容：

```
========== [前端] 登录成功 ==========
[前端] ✅ Token 已保存到 localStorage
[前端] 👤 用户: 用户名
[前端] ✅ st_token cookie 已设置
[前端] 🍪 当前所有 Cookies: st_token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
==========================================
```

**✅ 如果看到以上日志**：Cookie 设置成功，继续下一步  
**❌ 如果没有看到 st_token**：说明 Cookie 设置失败，检查浏览器设置

### 4️⃣ 进入控制台

登录成功后自动跳转到控制台页面。

**检查浏览器控制台输出**：应该看到：

```
========== [前端] 页面初始化 ==========
[前端] 👤 当前用户: 用户名
[前端] 🔑 localStorage token: 存在 (eyJhbGciOiJIUzI1NiIs...)
[前端] ✅ 已设置 st_token cookie
[前端] 🍪 当前所有 Cookies: st_token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
==========================================
```

### 5️⃣ 手动检查 Cookie

在开发者工具中：

1. **切换到 Application 标签页**
2. **左侧找到 Cookies**
3. **选择你的域名**（如 `http://localhost:3000`）
4. **查找 `st_token`**

**应该看到**：
- Name: `st_token`
- Value: `eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...`（长字符串）
- Path: `/`
- Expires: 一年后的日期

**❌ 如果没有 st_token Cookie**：
- 刷新页面（F5）再检查
- 如果还是没有，可能是浏览器阻止了 Cookie

### 6️⃣ 打开服务器日志

在**另一个终端窗口**中：

```powershell

cd e:\酒馆\公益云酒馆多开管理平台\ST-server

# 查看实时日志
npm run dev
# 或者如果服务器已经在运行，可以直接看终端输出
```

**保持日志窗口可见**，方便实时查看

## 🔎 常见问题诊断

### ❌ 问题 1: 前端日志显示 "st_token Cookie: ❌ 不存在！"

**原因**：Cookie 未设置成功

**解决方案**：
1. 检查浏览器是否阻止了 Cookie
   - 浏览器设置 → 隐私和安全 → Cookie → 允许所有 Cookie
2. 刷新页面（F5），检查是否自动恢复
3. 重新登录

### ❌ 问题 2: 服务器日志显示 "未提供 token"

**原因**：Cookie 没有被发送到服务器

**服务器日志示例**：
```
[Auth] ❌ 拒绝访问 /用户名/st/ - 未提供 token
[Auth] 📝 可用的 cookies: 无
[Auth] 💡 提示: 请确保用户已登录并设置了 st_token cookie
```

**可能的原因**：
1. **跨域问题**：Cookie 的 domain 不匹配
2. **路径问题**：Cookie 的 path 设置不正确
3. **浏览器安全策略**：浏览器阻止了跨站 Cookie

**解决方案**：
1. 检查访问地址和管理平台的域名是否一致
2. 如果使用 IP 地址访问，确保都是同一个 IP
3. 检查 Nginx 配置是否正确

### ❌ 问题 3: 服务器日志显示 "Token 验证失败"

**服务器日志示例**：
```
[Auth] ❌ Token 验证失败: invalid signature
[Auth] 错误类型: JsonWebTokenError
[Auth] 💡 Token 格式无效
```

**可能的原因**：
1. Token 已过期
2. Token 被篡改
3. JWT_SECRET 不一致

**解决方案**：
1. 重新登录获取新的 Token
2. 检查 `.env` 文件中的 `JWT_SECRET`

### ❌ 问题 4: 服务器日志显示 "权限不足"

**服务器日志示例**：
```
[Auth] ❌ 拒绝访问 /用户A/st/ - 权限不足
[Auth] 📝 当前用户: 用户B, 请求访问: 用户A
```

**原因**：用户 B 尝试访问用户 A 的实例

**解决方案**：只能访问自己的实例，或使用管理员账号

### ❌ 问题 5: Nginx 没有调用权限验证

**现象**：服务器日志完全没有 `[Auth Check]` 输出

**可能的原因**：
1. Nginx 配置中没有 `auth_request` 指令
2. Nginx 配置的路径不正确
3. Nginx 服务没有重启

**解决方案**：
1. 检查 Nginx 配置文件：
   ```nginx
   location /用户名/st/ {
       auth_request /auth-check/用户名;  # 这行必须存在
       # ...
   }
   ```

2. 检查 auth_request 端点：
   ```nginx
   location = /auth-check/用户名 {
       internal;
       proxy_pass http://st_manager/api/auth-check/verify/用户名;
       # ...
   }
   ```

3. 重启 Nginx：
   ```bash
   sudo systemctl restart nginx
   # 或
   sudo nginx -s reload
   ```

## 🎯 完整的成功流程

一个正常的访问流程应该是：

```
1. 用户登录
   ↓ [前端] Token 保存到 localStorage 和 Cookie
   
2. 进入控制台
   ↓ [前端] 自动设置 st_token cookie
   
3. 点击访问地址
   ↓ [前端] 打开新标签页，发送请求到 /{username}/st/
   
4. Nginx 拦截请求
   ↓ 发起 auth_request 到 /api/auth-check/verify/{username}
   
5. 服务器验证权限
   ↓ [后端] 从 Cookie 读取 st_token
   ↓ [后端] 验证 JWT Token
   ↓ [后端] 检查用户权限
   ↓ [后端] ✅ 返回 200 OK
   
6. Nginx 放行
   ↓ 转发请求到 SillyTavern 实例
   
7. 用户成功访问
   ✅ SillyTavern 界面正常显示
```

## 🛠️ 手动测试 Cookie

在浏览器控制台中执行：

```javascript
// 检查 localStorage
console.log('Token:', localStorage.getItem('token'));

// 检查 Cookie
console.log('All Cookies:', document.cookie);

// 检查特定 Cookie
console.log('st_token exists:', document.cookie.includes('st_token'));

// 手动设置 Cookie（如果需要）
function setCookie(name, value, days = 365) {
    const expires = new Date();
    expires.setTime(expires.getTime() + (days * 24 * 60 * 60 * 1000));
    document.cookie = `${name}=${value}; expires=${expires.toUTCString()}; path=/; SameSite=Lax`;
}

const token = localStorage.getItem('token');
if (token) {
    setCookie('st_token', token);
    console.log('✅ st_token cookie 已手动设置');
}
```
